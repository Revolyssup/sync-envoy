## Purpose

`sync-envoy` is a Go CLI tool for making **Istio-Envoy debugging easier**. It does two things:

1. **Syncs live cluster state to local files** — Istio CRs, XCP CRs, and Envoy sidecar configs are written to disk so you can inspect, diff, and edit them locally.
2. **Syncs local file edits back to the cluster** — drop a `_desired.yaml` file, the tool applies it to Kubernetes.

## CLI Commands

```
sync-envoy start      # Start all providers (blocks until SIGINT/SIGTERM)
sync-envoy stop       # Send SIGTERM to the running process (via /tmp/sync-envoy.pid)
sync-envoy cleanup    # Delete envoyconfigs/, istioconfigs/, xcpconfigs/
sync-envoy how        # Print colored explanation of how it works
sync-envoy pack       # Pack envoyconfigs/+istioconfigs/+xcpconfigs/ into .tar.gz
sync-envoy unpack <f> # Unpack .tar.gz back to the three dirs
```

### Key Flags (on `start`)
| Flag | Default | Description |
|------|---------|-------------|
| `--dir` | `envoyconfigs` | Where to write Envoy config JSON files |
| `--workload-selector` / `-w` | `""` | Label selector for Envoy pods (e.g. `app=httpbin`) |
| `--provider` | all | Comma-separated provider filter: `kubernetes,istio-file,envoy,xcp,xcp-file` |
| `--csds-address` | `""` | istiod gRPC address for CSDS streaming (e.g. `localhost:15010`) |
| `--log-level` | `info` | `debug`, `info`, `warn`, `error` |

## Architecture: Provider Pattern

Everything is built on a **Watcher → Event → Updater** pipeline defined in `pkg/types/types.go`:

```
Watcher --events--> chan types.Event ---> Updater
```

A `Provider` pairs one Watcher with one Updater and runs them concurrently. Multiple providers run in parallel via `provider.RunAll()`.

### Event struct
```go
type Event struct {
    Type     EventType         // EventAdd | EventUpdate | EventDelete
    Key      string            // "namespace/kind/name" or "kind/name"
    OldData  []byte
    NewData  []byte
    Metadata map[string]string // kind, name, namespace, pod_name, config_type, labels, ...
}
```

## The Five Providers

### 1. `kubernetes` — K8s CRs → files
- **Watcher**: `pkg/k8s/CRWatcher` — dynamic informers for 11 Istio CRD types:
  - `networking.istio.io/v1`: VirtualService, DestinationRule, Gateway, ServiceEntry, WorkloadEntry, WorkloadGroup
  - `security.istio.io/v1`: AuthorizationPolicy, PeerAuthentication, RequestAuthentication
  - `telemetry.istio.io/v1`: Telemetry
  - `extensions.istio.io/v1`: WasmPlugin
- **Updater**: `pkg/file/CurrentFileUpdater`
  - Writes `istioconfigs/<ns>/<kind>/<name>_current.yaml`
  - Skips write when content is unchanged (in-memory diff check)
  - With `WithTopology(topo)`: calls `topology.ExtractIstioEdges()` and updates `istioconfigs/topology.md`

### 2. `istio-file` — `_desired.yaml` → K8s cluster
- **Watcher**: `pkg/file/DesiredFileWatcher` — fsnotify on `istioconfigs/` recursively
  - Auto-renames plain `.yaml` files to `_desired.yaml`
  - Ignores `_current.yaml` files
- **Updater**: `pkg/k8s/CRUpdater`
  - Parses YAML → dynamic client Create/Update
  - Compares current cluster state, skips apply when identical
- **Mutual exclusion**: automatically disabled (with a warning log) when `xcp-file` provider is active, since xcpconfigs/ becomes the source of truth

### 3. `envoy` — Envoy sidecars → JSON files
- **Watcher**: `pkg/envoy/EnvoyWatcher` — tries three strategies:
  1. **CSDS gRPC streaming** via istiod (`--csds-address`): uses go-control-plane protobuf
  2. **Admin endpoint polling** (every 5s): `kubectl exec <pod> -- curl localhost:15000/config_dump`
  3. **istioctl polling** (every 5s, final fallback): `istioctl proxy-config <type> <pod> -o json`
- **Updater**: `pkg/envoy/FileUpdater`
  - Writes `envoyconfigs/<ns>/<pod>/<type>.json` (cluster, listener, route, endpoint, bootstrap, secret)
  - JSON wrapped in `TimestampedConfig{last_updated, pod_name, namespace, config_type, config}`
  - Ignores `last_updated` field when diffing (suppress timestamp noise)

### 4. `xcp` — XCP CRs → files + topology
- **Watcher**: `pkg/xcp/XCPWatcher` — dynamic informers for XCP CRDs across 6 API groups:
  - `xcp.tetrate.io/v2`: Workspace, WorkspaceSetting, Cluster, GlobalSettings, API
  - `traffic.xcp.tetrate.io/v2`: TrafficGroup, TrafficSetting, ServiceRoute, ServiceTrafficSetting
  - `gateway.xcp.tetrate.io/v2`: Gateway, GatewayGroup, IngressGateway, EgressGateway, Tier1Gateway, SharedGatewayReferenceGrant
  - `security.xcp.tetrate.io/v2`: SecurityGroup, SecuritySetting, ServiceSecuritySetting
  - `extension.xcp.tetrate.io/v2`: WasmPluginDefinition
  - `istiointernal.xcp.tetrate.io/v2`: IstioInternalGroup
- **Updater**: `pkg/xcp/XCPFileUpdater`
  - Writes `xcpconfigs/<ns>/<kind>/<name>_current.yaml`
  - Queries cluster to find Istio resources produced by this XCP resource (label-based then name-based)
  - Updates `xcpconfigs/topology.md`

### 5. `xcp-file` — `_desired.yaml` in xcpconfigs → K8s cluster
- **Watcher**: `pkg/file/DesiredFileWatcher` on `xcpconfigs/`
- **Updater**: same `pkg/k8s/CRUpdater` as the `file` provider

## File Layout

```
istioconfigs/
  <namespace>/
    <kind>/
      <name>_current.yaml    ← written by kubernetes provider
      <name>_desired.yaml    ← edit this to apply changes
  topology.md                ← Istio resource relationship map

envoyconfigs/
  <namespace>/
    <pod>/
      cluster.json, listener.json, route.json, endpoint.json, bootstrap.json, secret.json

xcpconfigs/
  <namespace>/
    <kind>/
      <name>_current.yaml    ← written by xcp provider
      <name>_desired.yaml    ← edit this to apply XCP changes
      <name>_expected_desired.yaml  ← for validation/comparison
  topology.md                ← XCP → Istio resource map
```

## Topology System (`pkg/topology/`)

Replaced the old correlation JSON approach. Two topology files are generated:

### `istioconfigs/topology.md`
Generated by `kubernetes` provider via `topology.ExtractIstioEdges()`. Shows relationships parsed from Istio YAML:

| Resource | What's extracted |
|----------|-----------------|
| VirtualService | `spec.gateways` → Gateway refs; `spec.http/tcp[].route[].destination` → hosts:ports |
| Gateway | `spec.selector` → workload labels; `spec.servers[].hosts` + port/protocol |
| DestinationRule | `spec.host`; `spec.subsets[].name` + labels |
| AuthorizationPolicy | `spec.selector.matchLabels` |
| PeerAuthentication | `spec.selector.matchLabels` |
| RequestAuthentication | `spec.selector.matchLabels` |
| ServiceEntry | `spec.hosts` |
| All (with XCP labels) | `xcp.tetrate.io/workspace` + group label → provenance edge |

### `xcpconfigs/topology.md`
Generated by `xcp` provider. Shows XCP → Istio resource mapping:

```
IngressGateway/e2e-test-gw ---> Gateway/e2e-test-gw (echo)
IngressGateway/e2e-test-gw ---> VirtualService/vs-echo-tetrate-io (echo)
```

Lookup strategy:
1. Label selector using XCP hierarchy labels (`xcp.tetrate.io/workspace`, `xcp.tetrate.io/gatewayGroup`, etc.)
2. Name-based convention fallback

### XCP → Istio Kind Mapping
```go
"ingressgateway":        → Gateway, VirtualService, EnvoyFilter
"egressgateway":         → Gateway, VirtualService
"tier1gateway":          → Gateway, VirtualService, EnvoyFilter
"gateway":               → Gateway, VirtualService, EnvoyFilter
"serviceroute":          → VirtualService, DestinationRule
"trafficsetting":        → Sidecar, EnvoyFilter, DestinationRule
"servicetrafficsetting": → DestinationRule
"securitysetting":       → PeerAuthentication, AuthorizationPolicy
"servicesecuritysetting":→ AuthorizationPolicy
```

### `topology.File` API
```go
topo := topology.NewFile(basePath, title)
topo.Set(ns, kind, name, []topology.Edge{...})  // upsert + rewrite file
topo.Remove(ns, kind, name)                      // remove + rewrite file
```

Edge format in markdown:
- With relation: `From --[relation]--> To`
- Without relation: `From ---> To`

## Diff Detection

All updaters track last-written state in memory. Before every write/apply:
- `diff.Compute()` — LCS-based unified diff for YAML/text
- `diff.ComputeJSON()` — JSON diff with configurable ignore paths (used to suppress `last_updated` timestamps in Envoy configs)
- Empty diff → skip write (logged as debug)
- Non-empty diff → write and log unified diff at INFO

## Key Implementation Details

### K8s Client (`pkg/k8s/client.go`)
- Uses in-cluster config or `~/.kube/config`
- Provides `Dynamic` (dynamic.Interface) and `Discovery` clients
- `CleanMetadata()` strips `status`, `managedFields`, `resourceVersion`, `uid`, `creationTimestamp`, `generation` before apply

### XCP Watcher metadata
The XCP watcher passes `labels` as a JSON string in `event.Metadata["labels"]` so the updater can build label selectors without a separate API call.

### Envoy config format
```json
{
  "last_updated": "2026-03-13T19:04:11Z",
  "pod_name": "httpbin-abc",
  "namespace": "default",
  "config_type": "cluster",
  "config": { ...raw envoy config dump... }
}
```

### PID file
Running process writes PID to `/tmp/sync-envoy.pid`. `stop` command reads it and sends SIGTERM.

## Things Removed / Historical Context

- **Correlation JSON** (removed in recent session): Previously the code wrote:
  - `envoyconfigs/<ns>/<pod>/<kind>-correlation.json` — which Envoy resources were shaped by which Istio CR
  - `istioconfigs/<ns>/<kind>/correlation.json` — which pods reference this Istio resource
  - `xcpconfigs/<ns>/<kind>/correlation.json` — which Istio resources this XCP resource produced
  - `istioconfigs/<ns>/<kind>/xcp-correlation.json` — reverse XCP index
  - These were deemed "meaningless" and replaced with the topology.md approach above
- The `pkg/correlation/` package was deleted entirely
- `pkg/xcp/correlation.go` was deleted entirely
